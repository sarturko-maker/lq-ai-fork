Knowledge bases: hybrid retrieval (lexical + vector)

How a query becomes grounded context for chat, skills, and citation verification. ↩ Learn source: retrieval.py ↗

Sample query

Each query fans into two retrieval arms over the same KB chunks, then the arms are merged. The lexical arm rewards literal term overlap; the vector arm rewards meaning.

Mix weight hybrid_alpha

Per-KB setting (default 0.5), overridable per query. 0 = vector-only, 1 = lexical/FTS-only. The fused score is (1 - alpha)·vector + alpha·fts.

Result count top_k

Each arm pulls top_k × 4 candidates (CANDIDATE_OVERSHOOT) before merge. Default 10, max 50.

FTS-only fallback

If the query embedding can't be produced (gateway down) or alpha = 1, the vector arm is skipped and ranking degrades to FTS-only — retrieval still works.

1 · Ingestion — document → chunks → tsvector + embedding → ready
Chunking is a character-precise sliding window (DEFAULT_TARGET_CHARS = 2000, DEFAULT_OVERLAP_CHARS = 200) that snaps boundaries back to the nearest sentence/paragraph break. The invariant content == normalized_content[char_offset_start:char_offset_end] is what the citation engine later verifies against. Each document_chunks row carries a generated content_tsv tsvector (for the lexical arm) and an embedding vector(1536) (for the vector arm). A file is only searchable once its files.ingestion_status = 'ready'.
2 · The query fans into two arms
query embedded → 1536-dim vector

Both arms scan document_chunks joined to documents → files → knowledge_base_files, filtered to files.ingestion_status = 'ready' and not soft-deleted, in this KB. Showing the top candidates per arm.

lexical Postgres full-text search

Stemmed, operator-safe; GIN-indexed tsvector.
rank = ts_rank_cd( dc.content_tsv, plainto_tsquery('english', :q) ) -- GIN index ix_document_chunks_content_tsv

vector pgvector cosine

Query embedding via the Inference Gateway.
score = 1.0 - ( dc.embedding <=> CAST(:q AS vector) ) -- <=> = cosine distance (pgvector) -- sequential scan; ANN index deferred
3 · Min-max normalize each arm, then linear-combine

Fused ranking — (1 - alpha)·vec_norm + alpha·fts_norm

Each arm's raw scores are min-max normalized to [0, 1] across the candidate union (not z-score — too few candidates at this scale), then combined with the alpha weight. A chunk found by both arms gets contributions from both sides. The top top_k by fused score are returned. These chunks are formatted into an LQ.AI Knowledge Context system block (numbered [n] markers) and handed to the model. That retrieval-context block is flagged lq_ai_skip_anonymization = True (Decision M2-1) so the gateway lets the model see intact source — and the same chunk text is what the citation engine verifies quoted spans against.
How this works in the codebase

Hybrid retrieval is M1 (shipped). The lexical arm is Postgres FTS — ts_rank_cd over a generated content_tsv tsvector, not a literal BM25 library. The vector arm is pgvector cosine distance (<=>). Fusion is a min-max-normalized weighted linear combination via hybrid_alpha — not Reciprocal Rank Fusion. The pgvector ANN index (ivfflat/hnsw) is deferred, so the vector arm is a sequential cosine scan today.
