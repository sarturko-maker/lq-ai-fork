What CI runs on every PR — and what it doesn't
Not in the per-PR CI matrix — verify these yourself
  • Cypress E2E (web/cypress/, 16 specs) — the Web job runs svelte-check + Vitest only; Cypress runs against a deployed stack, not in the unit CI job.
  • Repo-root tests/ (cross-cutting, e.g. test_observability.py, test_error_code_contract.py) — CI runs api/, gateway/, and web/ only. Run root tests with api/.venv/bin/python -m pytest tests/ -q.
  • Coverage gate — the 80% (api) / 90% (gateway) targets are tracked but not yet build-failing in CI (PRD §6, on the engineering-discipline roadmap).
  • provider-marked tests — hit a real LLM; require an API key, so they're skipped in CI. Run them locally with a key + -m provider.
Two rules that apply to every change (per CLAUDE.md): a bug fix ships with a regression test that fails before the fix, and a feature change ships with the doc update that describes it (the PRD section, the OpenAPI sketch, or the DB-schema doc). Tests and docs are part of the change, not a follow-up.
