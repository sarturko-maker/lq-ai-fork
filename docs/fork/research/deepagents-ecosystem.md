## 1. ADOPT NOW (F0-S9 and F1)

### 1.1 Version / pin actions

- **Stay on `deepagents==0.6.8`** — it is HEAD. Two pin tightenings, both one-liners:
  - **Raise the langgraph floor to `>=1.2.4`**: the subagent-interrupt-overwrites-parent-state bug (#2781) is only fixed with deepagents 0.6.7 + langgraph 1.2.4 together ([github.com/langchain-ai/deepagents/issues/2781](https://github.com/langchain-ai/deepagents/issues/2781)). We currently have 1.2.4 installed but the pyproject floor is `>=1.0` — a rebuild could silently regress.
  - **Code all new `StoreBackend` namespace factories new-style** (`lambda rt: (...)` receiving a langgraph `Runtime`): legacy ctx-style factories, `BackendContext`, and `StoreBackend(runtime=...)` are flagged "removed in deepagents==0.7.0" in 0.6.8 source (deepagents `backends/store.py:45-118`). Makes the eventual 0.7 bump a no-op.
  - Do **not** import `SubagentRunStream`/`AsyncSubagentRunStream`/`SubagentTransformer` — un-exported in 0.6.8 itself (release notes, [github.com/langchain-ai/deepagents/releases](https://github.com/langchain-ai/deepagents/releases)).

### 1.2 F0-S9 model qualification — concrete adoptions

- **`HarnessProfile` / `register_harness_profile()` (beta, in 0.6.8)** is the first-party seam for per-model tuning: base prompt/suffix, tool renames, excluded tools/middleware, extra middleware, general-purpose-subagent config ([docs.langchain.com/oss/python/deepagents/models](https://docs.langchain.com/oss/python/deepagents/models), [/profiles](https://docs.langchain.com/oss/python/deepagents/profiles)). **No MiniMax/Qwen/Kimi profile ships at 0.6.8** (built-ins: Anthropic + OpenAI-Codex only) — write a MiniMax-M3 profile ourselves. Evidence this matters: tool renames + prompt guidance moved GPT-5.3-Codex 33%→53% on a tau2-bench subset ([langchain.com/blog/tuning-deep-agents-different-models](https://www.langchain.com/blog/tuning-deep-agents-different-models)). **Consequence: qualify each model WITH its tuned profile — untuned scores are noise** (expect 10–20pt swings from profile alone).
- **Set `model.profile["max_input_tokens"]` on the gateway-injected ChatOpenAI per candidate.** `compute_summarization_defaults(model)` in 0.6.8 falls back to a **170,000-token trigger** when the model has no profile — past MiniMax-M3's real window, so compaction never fires and the run hard-overflows instead (container source inspection, `deepagents.middleware.summarization`). This is both a harness fix and an eval dimension ("does the candidate survive compaction?").
- **Reuse LangChain's own eval suites instead of inventing one**: `libs/evals` in the deepagents monorepo (`deepagents_evals` + `deepagents_harbor`) carries **118 evals across 8 categories** — tool_use (53, incl. BFCL v3), file_operations (13), memory (22, incl. MemoryAgentBench) — with `--eval-category` filtering, and LangChain's CI already runs MiniMax-M2.x/Qwen/Kimi/GLM/DeepSeek model sets on it ([github.com/langchain-ai/deepagents/tree/main/libs/evals](https://github.com/langchain-ai/deepagents/tree/main/libs/evals), EVAL_CATALOG.md / MODEL_GROUPS.md). The containerized run pattern is `libs/harbor` (Terminal Bench 2.0 wrapper, [langchain.com/blog/evaluating-deepagents-cli-on-terminal-bench-2-0](https://www.langchain.com/blog/evaluating-deepagents-cli-on-terminal-bench-2-0)). *(Minor cross-report wrinkle: the official report cites `libs/harbor`, the community report `libs/evals` containing `deepagents_harbor` — both paths exist in the monorepo; verify exact layout when wiring.)* No public MiniMax-M3 numbers exist on this harness anywhere — our eval generates novel data; reuse their categories so numbers are comparable.
- **Score the documented weak-model failure axes** (all from the issue tracker): (a) final-answer-after-tool-call completion (#1746), (b) malformed/empty tool-call rate (#417), (c) parallel tool-call id integrity (#3587), (d) compaction survival, (e) think→tool_call→tool_result→think loop integrity (#1630).
- **Harness-engineering precedent** for "fix the harness, vary the model": harness-only changes moved a fixed gpt-5.2-codex 52.8→66.5 on Terminal Bench 2.0 ([langchain.com/blog/improving-deep-agents-with-harness-engineering](https://www.langchain.com/blog/improving-deep-agents-with-harness-engineering/)). The three middlewares described (PreCompletionChecklist, LocalContext, LoopDetection) are directly portable `AgentMiddleware`s. `RubricMiddleware` (in 0.6.8) is a shipped self-grading building block for the eval's self-check phase.

### 1.3 Gateway conformance (prerequisite for S9 numbers being real)

Three streamed-tool-call conformance tests on the Inference Gateway, all sourced from closed-as-provider-side upstream issues:

1. **Every streamed tool-call delta must carry a non-empty, unique, stable id** — synthesize one if the provider omits it. Empty ids hard-fail deepagents' `task` tool ("Tool call ID is required for subagent invocation"); maintainers explicitly put the fix on the gateway ([github.com/langchain-ai/deepagents/issues/3587](https://github.com/langchain-ai/deepagents/issues/3587)).
2. **`ChatOpenAI(use_responses_api=False)`** when pointing at the gateway — newer langchain-openai defaults to the Responses API and yields `ValueError: No generations found in stream` against OpenAI-compatible endpoints ([#3190](https://github.com/langchain-ai/deepagents/issues/3190), [docs workaround](https://docs.langchain.com/oss/python/deepagents/code/configuration#compatible-apis)).
3. **Reasoning-content round-trip**: gateway must not strip think/reasoning fields from deltas AND must accept them echoed back on assistant messages with tool results — the documented failure class for reasoning models in multi-turn tool loops ([#1630](https://github.com/langchain-ai/deepagents/issues/1630), [litellm reasoning docs](https://docs.litellm.ai/docs/reasoning_content)). Test the full loop, not single calls.

### 1.4 F1 — durability, cancel, cockpit

- **`durability="sync"` on graph invocation** in the arq worker. Default is `"async"`, which can lose a checkpoint for a step whose side effects already ran ([docs.langchain.com/oss/python/langgraph/durable-execution](https://docs.langchain.com/oss/python/langgraph/durable-execution)). *(See contradiction appendix re `"exit"`.)*
- **Idempotency ledger keyed `(run_id, tool_call_id)` inside `guarded_tool_call`.** This is the central F1 design fact: langgraph resume is replay-based — nodes after the checkpoint **re-execute, including all tool calls** in a ToolNode, and deepagents does not wrap tools in langgraph `@task`. But because the AIMessage carrying `tool_calls` is checkpointed before the tools node runs, a replay carries the **same tool_call_ids** — so recording results at the chokepoint and returning recorded results on replay makes resume safe, at near-zero cost, in the one seam we already own (durable-execution docs; duplicate-execution confirmations [#6533](https://github.com/langchain-ai/langgraph/issues/6533), [#6626](https://github.com/langchain-ai/langgraph/issues/6626); duplicate-payment incidents [Medium](https://medium.com/@mehul_parmar/the-hidden-replay-risk-in-langgraph-how-durable-execution-can-burn-you-1d966141e71a), [crewAI #5802](https://github.com/crewAIInc/crewAI/issues/5802)). Until the ledger exists: orphaned runs are marked FAILED and resume is a user action — never auto-resume.
- **`max_tries=1` on the arq run-execution job.** arq is at-least-once; redelivery after a hard worker death would silently re-run the graph — the self-inflicted version of LangGraph Platform's own sweeper bug. The orphan sweep owns recovery, keyed off `agent_runs` heartbeats, not arq state ([arq-docs.helpmanual.io](https://arq-docs.helpmanual.io/), [arq #343](https://github.com/samuelcolvin/arq/issues/343)).
- **Lease/fencing in `agent_runs`** (`claimed_by`/`claimed_at`/fencing token checked at `guarded_tool_call`): LangGraph Platform's sweeper re-dispatches deepagents subagent tool calls running >~180s while the original still executes ([langgraph #7417](https://github.com/langchain-ai/langgraph/issues/7417), open), and OSS langgraph has **no coordination preventing two executors resuming one thread_id** ([diagrid.io analysis](https://www.diagrid.io/blog/checkpoints-are-not-durable-execution-why-langgraph-crewai-google-adk-and-others-fall-short-for-production-agent-workflows)). Heartbeat from inside the run loop (per streamed event), not per job — deepagents subagents run minutes inside one tool call.
- **Cancel = arq `Job.abort()`** (`allow_abort_jobs=True` raises `CancelledError` into the job coroutine), runner finalizer catches **`BaseException`** (CancelledError is not an `Exception`), settles `agent_runs.status=cancelled`, accepts loss of since-last-checkpoint state. Free bonus: a stop-flag check in `guarded_tool_call` gives cooperative between-tool-calls cancel ([forum 2600](https://forum.langchain.com/t/cancel-graph-run/2600), [forum 2538](https://forum.langchain.com/t/stopping-endpoint-for-deep-agents/2538)). Verify under test that subagent subgraphs actually stop on CancelledError ([langgraph #5682](https://github.com/langchain-ai/langgraph/issues/5682)).
- **HITL decision inbox maps 1:1 onto shipped API**: `interrupt_on={'tool': InterruptOnConfig(allowed_decisions=[...], when=...)}` → `result.interrupts[0].value.action_requests` → `Command(resume={'decisions':[...]})`; checkpointer required (we have AsyncPostgresSaver); `FilesystemPermission(mode="interrupt")` landed in 0.6.8 itself ([docs.langchain.com/oss/python/deepagents/human-in-the-loop](https://docs.langchain.com/oss/python/deepagents/human-in-the-loop)). Persist interrupt payloads as settled `agent_run_steps` rows.
- **v3 `stream_events` typed projections (beta) behind an adapter**: `.subagents` handles (name/status/own `.messages`) + `lc_agent_name` metadata are the cleanest source for tool-call/subagent SSE frames and map directly to `agent_run_steps.parent_step_id` ([docs.langchain.com/oss/python/deepagents/event-streaming](https://docs.langchain.com/oss/python/deepagents/event-streaming)). Beta status (0.6.8 un-exported internals) is exactly why the adapter layer between v3 and our AI SDK UI Message Stream v1 frames is mandatory, not optional.
- **Capture subagent identity at stream time — the checkpointer will never give it back**: open issue [#1355](https://github.com/langchain-ai/deepagents/issues/1355) confirms subagent transcripts are unrecoverable from checkpoints (no task-tool-call→subagent-run link). This is direct validation of our settled-rows design.
- **Langfuse self-hosted observability**: official deepagents integration via one LangChain `CallbackHandler` at the runner invoke site, tagged run_id/org_id; self-host ≥3.125.0; point `LANGFUSE_HOST` at our instance to keep gateway-only egress intact ([langfuse.com/integrations/frameworks/langchain-deepagents](https://langfuse.com/integrations/frameworks/langchain-deepagents)). Known bug: interrupt/resume fragments traces and marks interrupted tools ERROR ([langfuse #10962](https://github.com/langfuse/langfuse/issues/10962)) — don't alert on tool ERROR status.
- **`adelete_thread(thread_id)`** exists in langgraph-checkpoint-postgres 3.1.0 and deletes all three checkpoint tables — wire it into thread delete (already a Backlog item).
- **Practice-area config schema reference**: deepagents-cli's project layout (`agent.json` + `AGENTS.md` + `tools.json` + `skills/` + `subagents/` folders) is LangChain's own "agent config as data" answer — mirror its shape in the `practice_areas` DB entity ([pypi.org/project/deepagents-cli/](https://pypi.org/project/deepagents-cli/)). The SaaS deployment side of the CLI is irrelevant.

## 2. ADOPT AT F2 (memory) — per-component verdicts

All verdicts grounded in wheel-source inspection (deepagents 0.6.8, langgraph-checkpoint-postgres 3.1.0, langmem 0.0.30). Headline: **every ADOPT is in already-pinned packages; every BUILD is thin and sits behind seams we own.** The official long-term-memory docs describe **literally our F2 design** — `memory=[...]` + `CompositeBackend(default=StateBackend(), routes={'/memories/': StoreBackend(namespace=...)})`, org-level memory "typically marked read-only to prevent injection attacks" ([docs.langchain.com/oss/python/deepagents/long-term-memory](https://docs.langchain.com/oss/python/deepagents/long-term-memory)).

| # | Component | Verdict | Evidence / notes |
|---|---|---|---|
| 1 | Storage substrate | **ADOPT** `AsyncPostgresStore` | Ships in langgraph-checkpoint-postgres (already pinned — **zero new deps**); hierarchical tuple namespaces, jsonb `$eq/$gt/...` filter search, single `store` table. Key as `(org_id, level, id)`. |
| 2 | Path routing `/memories/{level}/` | **ADOPT** `CompositeBackend`, four longest-prefix routes | Longest-prefix-wins verified in `backends/composite.py`; ls/glob/grep remap prefixes correctly at ≥0.6.8 (prefix bugs #1654/#2782/#3436 fixed). Zero custom routing code. |
| 3 | Always-loaded level digests | **ADOPT** `MemoryMiddleware(sources=[...])` | Injects AGENTS.md-style files into the system prompt wrapped in `<agent_memory>` + untrusted-data guidelines (aligns with our prompt-injection rule); HTML comments stripped since 0.6.3. Source list computed per-run at agent-build time fits our DI. |
| 4 | Read-only company/practice | **ADOPT** `FilesystemPermission(operations=['write'], paths=['/memories/company/**','/memories/practice/**'], mode='deny')` + **BUILD** ~30-line read-only `BackendProtocol` wrapper | Permission enforcement is middleware-level (tool interception), not storage-level — custom tools holding a store handle bypass it, and **subagent `permissions` REPLACE the parent's entirely** (docs). The wrapper (write/edit return `WriteResult(error='read-only')`) is the storage-level backstop; `guarded_tool_call` stays the audit chokepoint. |
| 5 | Extraction (what to remember) | **BUILD** on our arq job | No framework fits. Mine langmem 0.0.30's MIT extraction/consolidation prompts and namespace-placeholder pattern as starting IP; model via gateway budget alias. |
| 6 | Consolidation/dedup | **BUILD** in the same arq job | LangMem's durable executor is LangGraph-Platform-only (SaaS); its local executor loses pending work on process death. arq is the durable self-hosted equivalent (per ADR-F003). |
| 7 | Proposal/approval ("system proposes, user owns", ADR-0013 D4) | **BUILD** thin | No framework surveyed (LangMem, Mem0, deepagents) ships an approval queue. Mechanism: proposed-state rows + inline keep/undo + optional `mode='interrupt'` on sensitive paths. Anthropic's ~93% prompt-approval-rate finding says batch review/undo beats per-write modals ([manthanguptaa.in/posts/memory_is_a_mistake/](https://manthanguptaa.in/posts/memory_is_a_mistake/)). |
| 8 | Within-chat compaction | **ADOPT** `SummarizationMiddleware` + `SummarizationToolMiddleware` as-is | Implements ADR-F003 layer 1 verbatim **including offloading full evicted history to the backend at `/conversation_history/{thread_id}.md`** — the layer-3 verbatim-recall substrate for free. `compact_conversation` tool for on-demand compaction. Hand-roll nothing. |
| 9 | Memory search | **ADOPT** `store.search(filter=...)`; add `IndexConfig` later | `IndexConfig.embed` accepts a callable → **embedding calls can route through our gateway**, preserving gateway-only egress. pgvector HNSW/IVFFlat optional and non-breaking; FTS-fallback per ADR-F003 until a tier-compliant embedding provider exists. |
| 10 | Checkpoint retention (F1 spillover) | **BUILD** arq sweep; evaluate `AsyncShallowPostgresSaver` | No OSS checkpoint TTL (Platform-only). Production data: ~93 rows/conversation; "90%+ of DB is expired checkpoints" ([tadeodonegana.com](https://tadeodonegana.com/posts/scaling-langgraph-postgres-checkpointer/), [support.langchain.com](https://support.langchain.com/articles/6253531756)). Store TTL exists but its sweeper is a manual `sweep_ttl()` call — put it in the arq sweep, not the built-in daemon thread. |

Two doc-vs-reality corrections for the F2 implementer: (a) docs' namespace examples use `rt.server_info.user.identity` — **`Runtime.server_info` is `None` outside LangSmith deployments**; key via `rt.context` (context_schema) on our self-hosted stack (langgraph `runtime.py` source). (b) `StoreBackend(namespace=None)` falls back to a deprecated all-users-share-storage default — always pass an explicit factory.

Ready-made accelerators: official persistent-memory example in `examples/` ([#2687](https://github.com/langchain-ai/deepagents/issues/2687)); donbr deployment gist with PostgresStore + CompositeBackend org/user scoping ([gist](https://gist.github.com/donbr/780c6c342a14b151011af8e83b8f48a6)); **DiTo97/deepagents-backends** (MIT, pushed 2026-06-10) — full-protocol Postgres/S3/Redis/Mongo backends to adopt or crib for the read-only wrapper and protocol-conformance edge cases ([github.com/DiTo97/deepagents-backends](https://github.com/DiTo97/deepagents-backends)). The `libs/evals` memory category (MemoryAgentBench, 22 evals) can then qualify memory behavior.

## 3. LANDMINES (each with mitigation)

1. **`PatchToolCallsMiddleware` permanently wedges threads on cancel-mid-tool-call — OPEN at 0.6.8** ([#3789](https://github.com/langchain-ai/deepagents/issues/3789), acked 2026-06-08, no fix released). Its `Overwrite(...)` sentinel type-erases across any JSON serialization boundary; every subsequent state read 400s. **Directly in F1-cancel's blast radius.** Mitigation: explicit cancel-mid-tool-call test in F1; consider excluding the middleware via `HarnessProfile.excluded_middleware`; keep settled DB rows as source of truth so a wedged langgraph thread is recoverable; watch for the fix and re-pin.
2. **Summarization 170k fallback for unprofiled models** (0.6.8 source). Gateway-injected MiniMax-M3 has no profile → compaction never fires → hard context overflow. Mitigation: set `profile["max_input_tokens"]` at model construction (§1.2).
3. **Empty `tool_call_id` from OpenAI-compatible streaming kills subagent fan-out** ([#3587](https://github.com/langchain-ai/deepagents/issues/3587)) — fix is owned by **our gateway** (synthesize ids). Mitigation: gateway conformance test before S9 (§1.3).
4. **Responses-API default breaks OpenAI-compatible endpoints** ([#3190](https://github.com/langchain-ai/deepagents/issues/3190)). Mitigation: `use_responses_api=False` everywhere we construct ChatOpenAI; decide later whether the gateway ever implements `/v1/responses`.
5. **Reasoning-content stripping degrades multi-turn tool loops** ([#1630](https://github.com/langchain-ai/deepagents/issues/1630)). Mitigation: gateway preserves/echoes reasoning fields; S9 tests the full think→tool→result→think loop.
6. **Resume replays the whole tools node — duplicate side effects** (durable-execution docs; [#6533](https://github.com/langchain-ai/langgraph/issues/6533)). Applies to HITL grant approve/resume cycles too, not just crashes. Mitigation: `(run_id, tool_call_id)` ledger in `guarded_tool_call` (§1.4); until then, FAILED-not-resumed policy. Avoid mixing interrupting and non-interrupting tools in one parallel batch ([#6626](https://github.com/langchain-ai/langgraph/issues/6626)).
7. **Default durability `"async"` loses checkpoints on hard crash** — widens the duplicate-execution window. Mitigation: explicit `durability="sync"` (but see Appendix conflict with `"exit"`).
8. **No coordination against two executors on one thread_id; sweepers can re-dispatch live runs** ([#7417](https://github.com/langchain-ai/langgraph/issues/7417) — LangChain's own platform shipped this bug against deepagents subagents specifically). Mitigation: lease/fencing token + in-loop heartbeats (§1.4); `max_tries=1` on arq run jobs.
9. **Subagent transcripts unrecoverable from checkpoints** ([#1355](https://github.com/langchain-ai/deepagents/issues/1355), open). Mitigation: already designed around — settled `agent_run_steps` + capture identity at stream time; never plan a feature that reads subagent history from AsyncPostgresSaver.
10. **Unbounded checkpoint growth is WONTFIX at the middleware level** ([#2876](https://github.com/langchain-ai/deepagents/issues/2876)); official remedy is langgraph 1.2+ DeltaChannel (claimed 10–100x reduction, 5.27GB→129MB; [deep-agents-0-6 blog](https://www.langchain.com/blog/deep-agents-0-6)) which had teething crashes through late May ([#3564](https://github.com/langchain-ai/deepagents/issues/3564)). Mitigation: retention sweep regardless; **verify whether DeltaChannel is actually active in our 0.6.8 + langgraph 1.2.4 install before relying on it** (reports disagree on this — see Appendix).
11. **CompositeBackend has ZERO route isolation by design** ([#2884](https://github.com/langchain-ai/deepagents/issues/2884) closed not-a-bug; no construction-time validation [#2928](https://github.com/langchain-ai/deepagents/issues/2928)). Two routes sharing a store namespace silently share storage. Mitigation: constructor-level assertion that the four routes' namespace tuples are distinct — we write it, upstream won't. Also note [#2048](https://github.com/langchain-ai/deepagents/issues/2048): routes are queried sequentially — 4 levels = 4 serial round-trips per ls/glob; keep mounted trees small.
12. **Subagent `permissions` field REPLACES the parent's entirely** (subagents docs). A subagent spec without re-declared deny rules can write company/practice memory. Mitigation: re-declare permissions on every subagent spec AND the storage-level read-only wrapper (§2 row 4).
13. **AsyncPostgresSaver `setup()` migrations are invisible to alembic and contain `CREATE INDEX CONCURRENTLY`** (requires autocommit; fails on fresh DBs otherwise — [langgraph #2570](https://github.com/langchain-ai/langgraph/issues/2570)). Mitigation: known footgun for the F1 worker rebuild path; the alembic-owned-DDL option (Tiendanube pattern) restores CI/CD visibility at the cost of tracking upstream's MIGRATIONS list — decide in the F1 plan, minding our no-host-alembic rule.
14. **Rejected-grant loops**: HITL "reject" caused infinite retry loops until Apr 2026 ([#2946](https://github.com/langchain-ai/deepagents/issues/2946), fixed in our pin) — but the same behavioral risk applies to `guarded_tool_call` denials: **a denied grant must return actionable guidance text to the model**, or MiniMax-class models loop. Mitigation: denial messages carry "what to do instead"; regression-test.
15. **Skills-on-first-turn bug class** ([#3235](https://github.com/langchain-ai/deepagents/issues/3235), fixed in pin) and **MCP pre-flight HEAD without auth headers** ([#3284](https://github.com/langchain-ai/deepagents/issues/3284), fixed May 2026) — both fixed at 0.6.8, both worth regression tests since practice areas load skills/MCPs dynamically; plan an interactive OAuth bootstrap for any OAuth MCP server (talon README).
16. **Recursion/context propagation into subagents**: subagents silently capped at 25 steps pre-fix ([#1698](https://github.com/langchain-ai/deepagents/issues/1698)); `context_schema` propagation issues ([#2797](https://github.com/langchain-ai/deepagents/issues/2797)). Mitigation: audit `recursion_limit` + context propagation when building per-area agents — our F2 namespace keying depends on `rt.context` reaching subagents.

## 4. REJECTED / NOT RELEVANT

- **LangMem (0.0.30)** — 7.5 months stale, pre-1.0, drags langchain-anthropic/openai/trustcall into the SBOM, writes memory with no approval seam (violates ADR-0013 D4), durable executor is Platform-only; mine its prompts only.
- **langgraph-api / LangSmith Deployment self-host** — license-key-gated closed source with `beacon.langchain.com` phone-home; violates no-SaaS ([docs](https://docs.langchain.com/langsmith/deploy-standalone-server)). Its Postgres-queue+Redis-pubsub shape validates our arq plan, nothing more.
- **deepagents-cli managed deployment** — LangSmith key + managed preview = SaaS; only the agent.json folder layout is borrowed (§1.4).
- **deep-agents-ui** — speaks LangGraph Server protocol (useStream + assistant_id), not AI SDK UI Message Stream; UX reference only ([github.com/langchain-ai/deep-agents-ui](https://github.com/langchain-ai/deep-agents-ui)).
- **CopilotKit/AG-UI integration** — second frame-taxonomy to validate our SSE frame types against; not a dependency (we own the wire).
- **ContextHubBackend / LangSmithSandbox backends** — LANGSMITH_API_KEY SaaS pulls; skip.
- **open_deep_research** — not built on deepagents (plain LangGraph workflow); wrong substrate.
- **vstorm full-stack-ai-agent-template** — its "DeepAgents" is their own pydantic runtime, not langchain-ai/deepagents.
- **Temporal/DBOS** — right-sized escape hatch noted for the record; the tool_call_id ledger buys most of the safety at zero dependency cost ([langchain.com/resources/langgraph-vs-temporal](https://www.langchain.com/resources/langgraph-vs-temporal)).
- **Redis Streams for worker→API streaming** — plain pub/sub suffices because settled rows + DB-tail fallback already make our SSE idempotent (lossiness only costs animation, ADR-F004); revisit only if replayable live deltas become a requirement.
- **langmem `SummarizationNode`** — superseded by deepagents SummarizationMiddleware.
- **Async subagents (preview)** — graph_id/Agent-Protocol-server oriented, "not plug-and-play" per independent review; in-process ASGI mode noted for F3, but F1 cancel must not depend on it; imprecise resume open ([#3008](https://github.com/langchain-ai/deepagents/issues/3008)).
- **deepagents-talon** — explicitly experimental (0.0.1, "subject to removal"), leans on LangSmith Fleet; readable design reference for F3 only.
- **deepagents-code** — official Claude-Code-analog; UX benchmark, not a component.
- **Store TTL daemon thread** — use manual `sweep_ttl()` from arq instead.
- **No open-source legal-domain deepagents project exists** (explicit search came up empty; Harvey/Legora/Spellbook closed) — first-mover confirmation, nothing to adopt.

## 5. IMPACT ON THE ROADMAP

**F0-S9 (model qualification) — reshape from "build an eval" to "assemble a rig".** The slice should become: (1) gateway conformance tests first (tool_call_id synthesis, `use_responses_api=False`, reasoning round-trip) — without these, every score is measuring gateway bugs, not models; (2) a MiniMax-M3 `HarnessProfile` + `model.profile["max_input_tokens"]` as part of the harness under test — qualification is per-(model, profile) pair, never bare model; (3) adapt `libs/evals` categories (tool_use, file_operations) plus in-house legal tasks rather than authoring an eval framework; (4) score the five documented weak-model failure axes (§1.2). MILESTONES.md's existing S9 language ("tool-call frames — never regex over text; schema validity ≈100%") already points this way; the change is adding the profile/conformance prerequisites explicitly and reusing LangChain's eval categories for comparability.

**F1 run-lifecycle durability (currently "arq + startup sweep settling orphaned runs") — needs three additions and one decision.** Additions: `durability="sync"` + `max_tries=1` + lease/fencing with in-loop heartbeats (the #7417 counterexample shows naive sweeps re-dispatch live deepagents subagent calls); the `(run_id, tool_call_id)` idempotency ledger inside `guarded_tool_call`, sharing a unique constraint with audit rows so dedup and audit stay consistent (this also touches blocker #6's one-inference-per-message assumption); a cancel-mid-tool-call regression test specifically for the open `PatchToolCallsMiddleware` wedge (#3789), with `excluded_middleware` as the fallback. Decision: resume policy — v1 marks orphans FAILED and makes resume a user action ("system proposes, user owns"); the ledger is what later makes the resume button safe. Checkpoint retention (Backlog item "Checkpoint-row retention/cleanup") should be pulled into this slice: the sweep job is being written anyway, and `adelete_thread` + a time-based DELETE are small.

**F1 Cockpit v0 / subagent tree / decision inbox — build on shipped primitives, behind an adapter.** The subagent tree slice should consume v3 `stream_events` projections (`.subagents`, `lc_agent_name`) mapped to our SSE v2 frames through an explicit adapter module — beta API, internals already churned within 0.6.x. The slice must capture subagent identity at stream time into `agent_run_steps` because #1355 means reload-from-checkpointer can never reconstruct it; this hardens the existing settled-rows decision into a constraint. Decision inbox v1 maps directly onto `interrupt_on`/`action_requests`/`Command(resume=...)` — no custom protocol needed; persist interrupt payloads as steps. Validate frame taxonomy against deep-agents-ui and CopilotKit AG-UI before freezing types.

**F1 practice_areas — adopt the agent.json shape; re-declare permissions per subagent.** The `practice_areas` schema slice should mirror deepagents-cli's agent.json/skills/subagents folder convention as its config vocabulary, and the per-area `create_deep_agent` slice must treat subagent specs as security surfaces: permissions REPLACE (not inherit), tools OVERRIDE, middleware does not inherit — area config generation must emit complete per-subagent declarations. MCP-per-area work inherits the OAuth-bootstrap note.

**F2 memory — scope shrinks materially; re-write the slice list around the scorecard.** The CompositeBackend/StoreBackend/MemoryMiddleware/SummarizationMiddleware plan in MILESTONES.md is confirmed as the documented first-party pattern with zero new dependencies; the slice list should drop any "build storage/routing/compaction" work and replace it with the thin BUILD items: read-only wrapper (~30 lines), namespace-distinctness assertion, proposed-memories approval flow, arq extraction/consolidation job (LangMem prompts as IP), and the two doc-corrections (rt.context keying, new-style factories). The runtime-isolation acceptance tests gain a specific adversary: a subagent spec with replaced permissions writing to `/memories/company/`. Backlog's "revisit Zep/Graphiti" stays parked — evidence says native is sufficient.

**F3 heartbeat — architecture pre-validated; keep arq, watch talon.** Upstream explicitly will not ship a scheduler (#3048: drive the compiled graph from any external scheduler) and talon demonstrates the same shape (external cron → graph, per-conversation serialization, explicit stop). The stateful-vs-stateless cron distinction from LangChain's runtime blog (heartbeat on a matter thread vs fresh thread per run) should be named in the F3 design, and heartbeat multiplies checkpoint volume — retention from F1 is a hard prerequisite.

## Appendix — cross-report contradictions and verify-first items

1. **`durability="sync"` (durability-ops) vs `durability="exit"`/`AsyncShallowPostgresSaver` (memory-state).** Real tension, not an error: `sync` maximizes crash-resume fidelity and minimizes the duplicate-execution window; `exit`/shallow minimize checkpoint bloat but sacrifice mid-run crash recovery (interrupts still persist under `exit`). Resolution belongs in the F1 plan: start with `sync` + retention sweep (correctness first); move to `exit`/shallow only if we formally adopt restart-over-resume and accept that settled DB rows are the sole mid-run history. Don't blend per-report advice silently.
2. **0.6.8 dependency floors differ**: official (PyPI JSON) says `langchain-core>=1.4.0, langchain>=1.3.4, langchain-anthropic>=1.4.3, langsmith>=0.8.3`; community (main-branch pyproject) says `>=1.4.6 / >=1.3.7 / >=1.4.5 / >=0.8.11`. Community likely read main, not the 0.6.8 sdist. Immaterial for us (installed: langchain 1.3.7) but resolve from the wheel METADATA before writing SBOM entries.
3. **DeltaChannel status in our install is unverified.** Durability-ops says "verify our 0.6.8 already uses it"; community notes it landed in langgraph 1.2.0a6 with crash bugs through late May (#3564); no report confirmed it is active in langgraph 1.2.4 + deepagents 0.6.8. Verify empirically (checkpoint row sizes on a long thread) before counting on the 10–100x claim.
4. **Eval package location**: `libs/harbor` (official) vs `libs/evals` containing `deepagents_evals` + `deepagents_harbor` (community). Both probably exist in the monorepo; confirm paths when wiring S9.
5. **Summarization-in-subagents vs checkpointer-in-subagents**: official confirms summarization middleware runs in every subagent by default, while memory-state's mitigation menu suggests compiling the checkpointer only into the top-level graph — and community's #1355 shows subagents are already compiled without checkpointers (which is why their transcripts vanish). Net: subagent state is ephemeral by construction; design F1's "how much work does an orphaned run lose" answer around whole-subagent granularity (one `task()` call = one all-or-nothing node, per #7417 evidence).
6. **Dates**: the official report self-describes fetching PyPI "2026-06-12" while its summary header says "June 2026" mapped at 0.6.8 — consistent; all four reports independently agree 0.6.8 (2026-06-03) is latest, so the pin verdict is high-confidence.
