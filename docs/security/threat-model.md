# Threat Model

> **Depth:** Summary-level. STRIDE-by-component for the 5 production services. Named threats + named mitigations with cross-references. Detailed design intent lives in PRD §5 and the ADR series.

## Trust boundaries

LQ.AI runs as 7 services on a single operator-controlled deployment (Docker Compose for dev; Helm/Kubernetes for production per `deploy/helm/lq-ai/`). The Inference Gateway is the only component holding plaintext provider API keys per PRD §4; this defines the primary trust boundary. Everything internal to the operator's deployment is one trust zone; the LLM providers (Anthropic, OpenAI, etc.) are another; the operator's IdP (if integrated) is a third.

```
┌────────────────────────────────────────────────────────────┐
│ Operator deployment                                        │
│  ┌──────┐   ┌─────────┐   ┌──────────┐   ┌────────────┐   │
│  │ web  │──>│   api   │──>│ gateway  │──>│ providers  │   │
│  └──────┘   └────┬────┘   └────┬─────┘   └────────────┘   │
│                  ▼             ▼                          │
│              ┌────────┐    ┌─────────┐                    │
│              │postgres│    │  minio  │                    │
│              └────────┘    └─────────┘                    │
│              ┌────────┐                                   │
│              │ redis  │                                   │
│              └────────┘                                   │
└────────────────────────────────────────────────────────────┘
```

The five rows in the STRIDE table below cover the production-facing services (`api`, `gateway`, `web`, `postgres`, `minio`). Redis and the ingest-worker are cluster-internal and inherit `postgres`-tier mitigations (least-privilege role, no external listener, operator-managed secret); they do not add a distinct row.

## STRIDE-by-component

| Component | Spoofing | Tampering | Repudiation | Info Disclosure | DoS | Elevation of Privilege |
|---|---|---|---|---|---|---|
| **api** | **Threat:** Forged or replayed JWT (e.g., obtained from leaked logs, MITM on a misconfigured reverse proxy, or developer-machine theft) lets an attacker impersonate the legitimate user. **Mitigation:** JWTs are HS256-signed against `JWT_SECRET` (ADR 0002); access tokens expire in ~15 min, refresh tokens rotate on use and are hashed at rest in `user_sessions`, and `must_change_password` plus admin-driven session revocation provide server-side invalidation. | **Threat:** Tampered request bodies (oversized fields, type confusion, mass-assignment) attempt to mutate fields the caller should not control. **Mitigation:** Every endpoint validates the request body through Pydantic models generated from `docs/api/backend-openapi.yaml`, and all mutating handlers re-derive `owner_id` from the authenticated principal rather than trusting client input. | **Threat:** A user denies having deleted a matter, exported a chat, or changed a tier-config; an admin denies tampering with the audit log itself. **Mitigation:** Every state-changing call writes an `audit_log` row with actor, action, before/after hashes, IP, user-agent, `privilege_marked`, and `routed_inference_tier` (PRD §5.3); admin actions on the audit log itself are flagged `privilege_marked` and cannot be silently dropped. | **Threat:** Stolen access token (XSS exfil from `web`, shoulder-surfed bearer header, leaked debug log) grants the bearer the user's data window. **Mitigation:** Short JWT TTL bounds the window; `RateLimited` (`api/app/errors.py`) gates the login surface; refresh tokens are hashed in `user_sessions` so DB read alone cannot resurrect a session; per-user data export (PRD §5.3) is RBAC-gated and audit-logged. | **Threat:** Unauthenticated request flood (login brute-force, expensive endpoints like file upload) starves the service. **Mitigation:** Authentication required on every non-public route via FastAPI dependency injection; login is edge-rate-limited (`api/app/security/passwords.py`); body-size and content-type ceilings enforced before handler dispatch. | **Threat:** A Member or Viewer reads or mutates another tenant's data (chats, files, projects) by guessing resource IDs. **Mitigation:** Every query is scoped `WHERE owner_id = current_user.id` (see `api/app/api/files.py`, `chats.py`, `projects.py`); admin/member/viewer roles are an enum enforced via FastAPI dependencies (PRD §5.2); ADR 0005 keeps tenancy at the application layer rather than the storage key. |
| **gateway** | **Threat:** Forged inference request from inside the deployment network attempts to bypass the api and call a provider directly (skipping tier-floor + audit). **Mitigation:** Every gateway route requires the `X-LQ-AI-Gateway-Key` shared secret compared via `hmac.compare_digest` (`gateway/app/api/dependencies.py`); the key lives only in the gateway's and api's process environment. | **Threat:** A compromised admin path (or a malicious operator with config write access) tampers with `gateway.yaml` to silently re-route requests to an attacker-controlled provider. **Mitigation:** Atomic temp-file + `os.replace` writes with Pydantic validation on reload reject malformed configs without flipping the holder (ADR 0010); hardened deployments mount `gateway.yaml` read-only and forfeit the admin write affordance; alias edits are audit-logged at the api boundary. | **Threat:** Disputes about which provider/model served a privileged request (e.g., a Tier-3 commitment was actually fulfilled by a Tier-4 fallback). **Mitigation:** Every call writes an `inference_routing_log` row (`api/app/models/inference.py`) with `routed_provider`, `routed_model`, and `routed_inference_tier`; the same fields are persisted on the `Message` row and surfaced in the chat UI per ADR 0011, so the user-visible answer matches the audit record. | **Threat:** Provider API keys exfiltrated from `gateway.yaml`, environment listing, or a backup snapshot. **Mitigation:** ADR 0011 + `docs/security/encrypted-keys.md` ship the Fernet-wrapped `api_key_encrypted` path; plaintext exists only in `GatewayConfig.providers[*].api_key` in heap for the duration of the adapter call and is never logged, serialized, or returned by the admin API. | **Threat:** Token-cost amplification — an attacker (or runaway client) drives expensive provider calls to exhaust the operator's spend cap. **Mitigation:** PRD §5.5 cost tracking tags every request with user/capability/chat; per-tier and per-user quotas attach at the gateway router; the api enforces tier-floor refusal (D1) so a forged "downgrade" cannot land. | **Threat:** Tier escalation — a request asks for a Tier-1 model on a Tier-3-floored skill, or vice versa, to leak privileged content to a lower-assurance provider. **Mitigation:** Tier-floor enforcement runs server-side in the gateway (D1, ADR 0011); the resolved tier on every alias is published to the picker so users cannot silently land below a skill's `minimum_inference_tier`; admin alias edits do not bypass the floor check. |
| **web** | **Threat:** Session theft via XSS, malicious browser extension, or stolen `localStorage` lets an attacker reuse the LQ.AI shell's tokens. **Mitigation:** Tokens live under dedicated `lq_ai_access_token` / `lq_ai_refresh_token` keys distinct from OpenWebUI's (ADR 0009); the LQ.AI shell calls the api over HTTPS with `Bearer` headers (not cookies, so the surface is XSS-bound rather than CSRF-bound); the api's short TTL + refresh rotation bounds the stolen-token window. | **Threat:** XSS via skill markdown, model output, or chat content tampering the DOM and exfiltrating tokens. **Mitigation:** `MessageBubble.svelte` renders assistant content through `marked` + `DOMPurify.sanitize`; SvelteKit's default escaping bounds non-message paths; the OpenWebUI shell at `/` is intentionally isolated from the LQ.AI shell's state per ADR 0009. | n/a (web does not author audit events; every state-changing action is the api's audit row, which records actor, IP, and user-agent regardless of which shell originated it). | **Threat:** Disclosure of cached skill/file content from `localStorage` or service-worker storage when a shared device is repurposed. **Mitigation:** No provider keys, JWTs of other users, or raw file bytes are ever written to `localStorage`; the LQ.AI client clears its token pair on logout and on the 401-then-refresh-fail path (ADR 0009 §"Token refresh"); operators with stronger requirements front the deployment with an idle-timeout reverse proxy. | **Threat:** Client-side render bomb (multi-megabyte assistant message, runaway markdown nesting) freezes the browser tab. **Mitigation:** The api enforces message-content size ceilings before persistence; the streaming SSE parser (`web/src/lib/lq-ai/sse/parser.ts`) emits bounded `MessageDelta` frames and the consumer caps in-flight chunks. | n/a (the browser sandbox bounds privilege escalation to the user's own session; cross-origin isolation between the LQ.AI shell and the OpenWebUI shell follows the standard same-origin model per ADR 0009). |
| **postgres** | n/a (cluster-internal; not exposed outside the operator's deployment network — same posture as the Compose/Helm reference topologies in PRD §2.4). | **Threat:** Row tampering by a co-located service with broader write rights than it needs, or by an attacker who has pivoted into the deployment network. **Mitigation:** The api connects as a non-superuser application role per PRD §5.6; pgcrypto column encryption protects the audit-log payload and `api_key_encrypted`-shaped fields; backups are operator-encrypted at the infrastructure layer. | n/a (Postgres server logs are operator-managed and outside the application's audit trail; the application's repudiation story lives in the `audit_log` table itself). | **Threat:** Cross-tenant disclosure via `pgvector` similarity search returning embeddings whose document the requesting user does not own. **Mitigation:** Every vector query joins through the `files`/`documents` tables and re-applies `owner_id`-scoped predicates (the same posture as the per-handler `WHERE owner_id` filter in ADR 0005); embeddings without an owner row are unreachable from the api. | **Threat:** Connection exhaustion (leaked sessions, runaway report query) starves all callers. **Mitigation:** SQLAlchemy connection pool ceiling in `api/app/config.py`; `/health` and `/ready` probes surface saturation; Helm chart sets resource requests/limits on the api Deployment so a single replica's pool exhaustion does not propagate. | **Threat:** Superuser escalation if the application role is over-privileged or shares credentials with the bootstrap role. **Mitigation:** Operators deploy with a dedicated, non-superuser app role (the Helm chart's default); Alembic migrations run as a separate higher-privileged role only during the deployment's migration job, not at request time. |
| **minio** | n/a (cluster-internal; the api is the only client of the object store per ADR 0005, so external spoofing reduces to spoofing the api). | **Threat:** Object tampering — a file's bytes are replaced after upload to subvert downstream skill/Citation-Engine output. **Mitigation:** Every upload is content-addressed by SHA-256 stored in `files.sha256` (ADR 0005); the api re-verifies the digest on download paths that touch privileged Projects; soft-delete preserves the original bytes for audit-log lookups. | **Threat:** Disputes about who uploaded what bytes (especially for privileged Projects). **Mitigation:** Every `POST /api/v1/files` writes an `audit_log` row with actor, file UUID, SHA-256, and `privilege_marked` per PRD §5.3 + §3.11; soft-delete keeps the row queryable so the audit trail resolves to a filename rather than a bare UUID (ADR 0005). | **Threat:** Cross-tenant object access if an attacker reaches the MinIO console with read credentials, or if an api bug leaks raw `storage_path` values to another user. **Mitigation:** ADR 0005's bare-UUID key scheme keeps no tenant identifier in the storage layer (so console-enumeration does not partition by user); the canonical tenancy boundary is the api's `WHERE owner_id = current_user.id` filter applied before any content read; root credentials live in an operator-managed Kubernetes Secret. | **Threat:** Storage exhaustion (large uploads, soft-deleted files that never reap) fills the bucket. **Mitigation:** Per-bucket quotas configured by the operator at deploy time; PRD §5.6 documents storage as operator-encrypted infrastructure; the GDPR Article 17 hard-delete path (D6, referenced by ADR 0005) is the load-bearing reaper. | **Threat:** MinIO admin escalation via a leaked root credential lets an attacker bypass the api's owner-scoping and read every file. **Mitigation:** Root credentials are only in the operator-managed Secret consumed by the MinIO StatefulSet; the api connects with a least-privilege access-key/secret-key pair scoped to the application bucket; admin console exposure is operator-controlled (typically not Internet-reachable). |

## Out-of-scope threats

- **Compromise of the operator's host / OS / supervisor.** LQ.AI assumes the operator's deployment substrate is trusted. Container escape, OS-level compromise, and physical access are operator responsibilities.
- **Compromise of an LLM provider.** Gateway minimizes blast radius (Anonymization, Tier Derivation, per-provider isolation), but a compromised provider that already received unredacted content is outside our trust envelope.
- **Compromise of operator-controlled secrets.** If `JWT_SECRET`, `LQ_AI_GATEWAY_KEY`, or `LQ_AI_GATEWAY_MASTER_KEY` leaks, an attacker can forge tokens, spoof internal calls, or recover provider keys. Operators rotate per their own runbooks; the gateway-config-hot-reload path (ADR 0010) and the master-key rotation procedure in `docs/security/encrypted-keys.md` support rotation without invasive downtime.
