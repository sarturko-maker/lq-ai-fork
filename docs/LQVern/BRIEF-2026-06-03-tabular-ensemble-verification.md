## The goal (from Donna)
A per-column **Ensemble verification** toggle. When `ColumnSpec.ensemble_verification` is true, that column's cells should be verified through the Citation-Engine **ensemble** cascade (multiple judge models → `verification_method ∈ {ensemble_strict, ensemble_majority}`) instead of the default, and `POST /tabular/preview-cost` should estimate **higher** for ensemble columns. The column flag is the per-column override of the skill-level `ensemble_verification`; honor the column value, fall back to skill/deployment default when null. Acceptance: in a completed execution, an ensemble column's cells carry an `ensemble_*` verification signal while a non-ensemble column on the same run does not; and preview-cost returns a higher estimate for ensemble columns. No change to the `skill_name` XOR `columns` contract; `minimum_inference_tier` unchanged.

## ⚠️ Premise corrections (verified against the code — read before designing)
1. **Tabular cells run NO Citation-Engine verification today.** `api/app/tabular/nodes.py` (~298–320) parses `cited_chunk_indices → cited_chunk_ids` and stores them (display-only refs — this is DE-309). There is **no "default" single-pass verification to swap for ensemble** — you are **wiring citation verification into the tabular cell path for the first time**, ensemble-mode. This overlaps DE-309 (resolvable Citation-Engine rows for tabular) and the just-shipped navigable-citation enrichment (#5, `_enrich_cell_citations` in `api/app/api/tabular.py`, and the `Citation` model in `api/app/schemas/tabular.py` which now carries `source_file_id`/`source_page`/`source_text`).
2. **Ensemble is a BACKEND citation-engine step, NOT a gateway request parameter.** The ask suggests adding an ensemble field to `schemas/gateway.py::ChatCompletionRequest` — **don't.** Chat doesn't do that. The model-extraction call to the gateway is unchanged; verification happens **after**, backend-side, by calling `app.citation.verify(...)` which itself makes judge calls *through* the gateway. So no `ChatCompletionRequest` schema change is needed.

## The reuse surface (this is the machinery to reuse — do NOT reimplement)
- **`api/app/citation/verification.py::verify(candidate, document, *, gateway, judge_model, ensemble_config) -> VerificationResult`** (~line 545). When `ensemble_config` is supplied, **Stage 4 (ensemble) replaces Stage 3** (per M2-D1 decision B) and the result's `verification_method` is `ensemble_strict`/`ensemble_majority`. The judges run via the `gateway` arg (`_JudgeGatewayProtocol`).
  - `candidate` is `_CandidateProtocol` (verification.py:67): needs **`source_text: str`** (the quoted span being verified) + `source_document_id` (+ offsets). `document` is `_DocumentProtocol` (line 80) with `normalized_content`.
- **`api/app/clients/gateway.py::GatewayClient.get_citation_engine_ensemble_config() -> EnsembleConfig | None`** (line 266) — the resolved Stage-4 config (judge count/models, strict-vs-majority rule). Chat fetches it and passes it to `verify`.
- **Chat's call site to mirror:** `api/app/api/chats.py:1872` — builds `candidates` (each with `source_text` = the emitted quote + `source_document_id`), looks up `docs_by_id`, and loops `verify(cand, doc, gateway=…, judge_model=…, ensemble_config=…)`. Absence of a verified result = "unverified" (no row).

## The ONE design decision to lock at kickoff (the crux)
`verify()` is built around a **quoted span** (`candidate.source_text`) — chat citations quote verbatim text the model emitted, and Stages 1–3 check that span literally/tolerantly/semantically appears in the source. A **tabular cell has an extracted *value* + chunk refs, not a verbatim span.** So decide what `candidate.source_text` (the "claim") and `document` are for a tabular cell. **Recommended interpretation:** treat the **cell's extracted `value`** as `candidate.source_text` (the claim) and each **cited chunk's text** as the `document` (`normalized_content`), so the ensemble judges affirm *"does this chunk support this cell value?"* — a semantic judgment, which is exactly what Stage-4 ensemble does (and why it's the right stage for tabular, where the value rarely appears verbatim). Confirm this against the actual `_CandidateProtocol`/`VerificationResult` shapes and the ensemble judge prompt (`api/app/citation/` — `test_ensemble.py`, `test_paraphrase_judge.py`, the judge-prompt module) before building. If the ensemble judge prompt assumes a verbatim quote, you may need a tabular-appropriate judge framing — flag it.

## Scope (4 parts)
1. **`api/app/tabular/nodes.py` — verification wiring.** After a cell's `cited_chunk_ids` are resolved, when the column's effective `ensemble_verification` is true (column value, falling back to the skill/deployment default when null — find where the skill-level flag is resolved), fetch `ensemble_config` once per execution and, per cited chunk, call `verify(...)` with it. This adds async judge calls inside the LangGraph extraction/verification node — mind concurrency + the existing R4/cost + per-cell error handling (a judge failure must not crash the cell; degrade like chat does). Non-ensemble columns: unchanged (no verify call) — or, if you choose to also run *default* verification for them, that's a scope expansion → flag it; the ask only requires ensemble columns to be verified and non-ensemble columns to visibly differ.
2. **Surface the result on the cell.** Reflect the verification on the cell so it's observable — e.g. the cell's `confidence` and/or its citations' `verification_method`. The cell/citation shape lives in `api/app/tabular/state.py` (`_ColumnSpecState`, cell snapshot) + `api/app/schemas/tabular.py` (the `Citation` model — already carries `source_*` from #5; add `verification_method: str | None`). Persisted cell shape may need the field too (check `state.py` + how cells serialize). **No migration if the data rides in the existing JSON results blob** (like cited_chunk_ids does) — prefer that; if a column is genuinely needed, STOP and raise it.
3. **`api/app/tabular/cost.py` — ensemble cost premium.** `estimate_tabular_execution_cost` (~165) currently costs `per_cell_cost * cells_count` from calibrated metrics. Add a premium for ensemble columns: per ensemble-column cell, add the judge-call cost (reuse `app.citation.cost.estimate_judge_call_cost_usd` × judge count from `ensemble_config`), or a documented flat multiplier for v1. Surface the premium in the `preview-cost` response so the UI can show it (extend the response schema + `backend-openapi.yaml`). Columns must NOT be costed identically ensemble vs not.
4. **Contract + tests.** Update `docs/api/backend-openapi.yaml` for any response-shape change (cell `verification_method`, preview-cost premium). Keep `test_openapi` path count correct (no new path expected → stays 114). Tests: a completed execution with one ensemble column + one non-ensemble column → ensemble cells carry an `ensemble_*` signal, non-ensemble cells don't; preview-cost higher for ensemble. Use the throwaway pgvector DB; the ensemble judge calls need a gateway double/mock (mirror how chat's verification tests stub the judge gateway — see `tests/citation/test_ensemble.py` / `test_verify_cascade.py`).
